Multi-agent audit (6 reviewers over disjoint module groups, every finding adversarially verified by an independent agent reading the code, 3 approach panels). 58 findings raised; 57 confirmed (52 unique — the fusion-gate issue was independently confirmed by 4 reviewers), 1 refuted.

## Confirmed bugs

### [CRITICAL] src/pipeline.py:2014 — Anchor-primary placement is never reported as the primary position

The [10a*] block (lines 1863-1880) computes the anchored trajectory and its own comment says 'report THAT anchored start as the primary answer', but it only writes result["anchored_position"]. Section 11 then builds the headline via build_position_report(candidates[0], ...) at line 2014 using the UN-anchored candidate, so result["position"] (latitude/longitude, route_latlon, google_maps_url, printed summary, the hypotheses block, and the OrienterNet refinement prior at line 2081) all describe the raw matcher pick. Every consumer in the repo reads result["position"] (scripts/test_traffic_signal_discrim.py:103, scripts/viz_clips_where.py:43, scripts/viz_new_clips.py:42); none reads anchored_position.

**Failure scenario:** Ulm run with --use-vpr-prior: anchored start error 236 m is written to the side-field anchored_position, while result["position"].latitude/longitude and the printed 'ESTIMATED POSITION' block carry the matcher pick at 664 m (Erbach: ~7 km instead of 2.26 km). Any script or user consuming the standard position key gets the pre-anchor answer, silently discarding the project's main accuracy win.

### [CRITICAL] src/visual_odometry.py:252 — _fit_plane_projection PCA has arbitrary handedness — matcher input is mirrored on ~50% of clips

`axes_2d = Vt[:2].T` takes the top-2 SVD axes with numpy's arbitrary sign convention, so the projected 2D path can be a REFLECTION of the true top-down view (left turns become right turns). This is the DEFAULT path: `estimate_trajectory` defaults `enforce_planar=True` and pipeline.py:658 calls it with defaults, so `traj.xz` (the matcher input whenever no OpenVO file is staged) is mirrored whenever the SVD sign lands wrong. Downstream is strictly chirality-sensitive: the stage-1 filter is a Pearson correlation of heading-delta signatures (trajectory_matching.py:183, sign flips under mirror, so the true walk gets a NEGATIVE corr and is cut from the top-500), and both `procrustes_similarity` (skimage, no reflection) and `procrustes_fixed_scale` (explicit det<0 flip at trajectory_matching.py:122) forbid reflections. Verified empirically: on synthetic L-turn trajectories with random yaw, 7/20 came back mirrored vs the raw x-z projection (bearing-signature turn sign flipped). Fix: after SVD, enforce view-from-above handedness — the raw x-z drop has v1×v2 = -ŷ (camera +y is down), so flip `axes_2d[:,1] *= -1` when `np.cross(axes_2d[:,0], axes_2d[:,1])[1] > 0`.

**Failure scenario:** Any clip whose VO centers give SVD axes with flipped handedness (data-dependent coin flip, reproduced 7/20 synthetically): every turn direction in traj.xz is inverted -> the true route's bearing correlation goes negative and it is excluded from the stage-1 top-k -> the matcher can only return mirror-compatible (wrong) streets, hundreds of meters to km off.

### [MAJOR] scripts/run_all_gt.py:64 — Benchmark sweep reports stale result.json when a run fails (rc never gates the read; final table omits rc)

After `subprocess.run(...)`, the script reads `output/<slug>/result.json` whenever the file exists (line 64), regardless of the return code captured at line 60-61. main.py only writes result.json on success, so a crashed run leaves the PREVIOUS run's file in place, and its numbers are loaded into the row. The per-clip progress line prints rc, but the FINAL RESULTS table (lines 79-88) prints only mean/start/conf/spread/streets — rc is dropped, and 'FAIL' is shown only when gt_mean is absent, which a stale file always provides. Fix: delete/rename result.json before launching each clip, or require rc==0 AND result.json mtime > run start before reading.

**Failure scenario:** A code change breaks the pipeline on KITTI drive_0033 (rc=1, no new result.json written). Yesterday's result.json from a good run remains. The sweep's final table prints yesterday's mean=143 m as the current pipeline's accuracy — the regression is completely masked, defeating the script's stated purpose as a regression/accuracy sweep.

### [MAJOR] scripts/test_vpr_prior_accuracy.py:97 — 'Distance to GT route' is distance to the nearest sparse waypoint — quantization noise is the same magnitude as the differences being compared

`route_dist` (lines 96-97) takes min distance to the 10 GT WAYPOINTS, not to the route polyline. In ground_truth/ulm_ULl8s4qydrk.json consecutive waypoints are up to ~540 m apart (t=300->360 is ~537 m; t=360->415 is ~514 m). A prior sitting exactly ON the route midway between two waypoints reads as ~250-270 m 'to route'. The script's whole purpose is deciding whether MegaLoc regressed the prior from ~53 m to ~141-360 m and locking in the best model x aggregation — decisions made at exactly the scale of this measurement error, so configs can be mis-ranked and the 'regression' partly an artifact of where each prior falls relative to waypoint positions. Fix: point-to-segment distance over consecutive waypoint pairs (the exact helper exists: src/evaluator.py `_segment_to_polyline_distance`), computed in a local metric projection. Secondary fidelity gap: queries are 80 frames sampled across the WHOLE video (lines 87-90), while production `kartaview_vpr_prior` uses n_query=40 frames drawn only from the analyzed vo-segment (default 0:420) — the study measures a different prior than the pipeline computes.

**Failure scenario:** Config A's prior lands 20 m from the true route but midway between waypoints (reported ~250 m); config B's prior is 100 m off-route but adjacent to a waypoint (reported ~100 m). The table declares B the winner and B gets locked in as the anchor-primary prior, degrading the headline placement accuracy the whole anchor-primary path depends on.

### [MAJOR] src/bev_splat_match.py:502 — Camera intrinsics passed at full-frame resolution while the ground image is resized to 256x1024 — K is wrong by ~2-8x per axis

`_run` resizes the ground image to (_GRD_H,_GRD_W)=(256,1024) (line 508) but passes `intrinsics` unmodified (line 502). The comment at lines 481-483 claims the model 'internally rescales them based on grd_depth.shape', but the upstream code (third_party/BevSplat/models/models_kitti_nips.py:535-536) DIVIDES K rows by grd_depth width/height — i.e. it normalizes assuming K is already expressed in 256x1024-image pixels (ori_grdH/W are read from the input tensor at line 445, and World2GrdImgPixCoordinates line 144 makes the same assumption). For a 1080p frame with default 70-deg intrinsics (fx=fy=1371, cx=960, cy=540) the model sees normalized fx'=1.34 (should be 0.71), cx'=0.94 (should be 0.5), fy'=5.36 (should be ~1.27), cy'=2.11 — a principal point 2.1 image-heights below the image. Fix: scale row 0 of K by 1024/w_img and row 1 by 256/h_img before building K_t (note the resize is anisotropic, so fx and fy need different factors).

**Failure scenario:** --enable-bev-splat on any 16:9 dashcam clip: the ground-Gaussian unprojection uses a focal ~4-8x too large and an off-image principal point, so even with correct depth the g2s footprint lands in the wrong place at the wrong scale on every satellite tile; per-candidate scores are geometric nonsense yet still fused into consensus.

### [MAJOR] src/bev_splat_match.py:509 — All-zero depth placeholder collapses every BevSplat ground Gaussian to the camera center, making the channel structurally non-discriminative

`grd_depth = torch.zeros(1, _GRD_H, _GRD_W)` (line 509; same at 518 for the seq variant) is documented as a benign placeholder, but in the upstream encoder (third_party/BevSplat/gaussian/encoder_feat_nips.py:156) Gaussian means are computed as `means = origins + directions * grd_depth` — depth is the geometry, not a prior. With zeros, every Gaussian sits at the camera origin, the ground->satellite projected footprint degenerates to ~a pixel, the coverage mask in the score extraction (lines 661-677) is near-empty, and the score falls to the 0.5 neutral fallback or a 1-pixel cosine — exactly matching the recorded 0.5947-for-9/10-candidates behavior. Depth is also concatenated as an input feature (`grd_depth/20.0`, encoder line 137), so the network additionally runs out-of-distribution. The repo already has monocular depth (DA3) that could be resized into this slot.

**Failure scenario:** --enable-bev-splat with valid KITTI_no_GPS.pth weights and esri tiles: every candidate scores ~0.5±noise regardless of location; bev_rank becomes a noise permutation of near-equal scores and is fused with weight 0.75 into the top-5 consensus (pipeline.py:1734), reordering the shortlist on numerical noise rather than appearance.

### [MAJOR] src/da3_reconstruction.py:245 — Chunk-alignment picks a shared keyframe index outside the previous chunk — multi-chunk DA3 runs are silently stitched with identity transforms

The shared frame between chunk ci and ci-1 lies in [s, prev_e), but the code computes `shared_idx_in_prev = e - chunk_overlap`, an index near the CURRENT chunk's end. For every full-size follow-up chunk (step > overlap, i.e. whenever n_keyframes > batch_size), e - chunk_overlap >= prev_e, so `keyframe_indices[shared_idx_in_prev]` was never stored in `extr_global_per_keyframe` and the code falls into the 'no reliable shared keyframe' branch (lines 248-251): R_align=I, t_align=0. Concretely with the function's own defaults (n_keyframes=48, batch_size=32, chunk_overlap=4): chunks (0,32) and (28,48); shared_idx_in_prev = 48-4 = 44, but only keyframes 0..31 are stored -> chunk 2 is appended UNALIGNED, superimposed at the origin in its own world frame, and its poses even overwrite the overlap keyframes 28..31 (line 292). The resulting trajectory folds back to the origin mid-route and the point cloud is double-imaged. The pipeline's `_da3_trajectory_plausible` reversal guard then rejects the whole DA3 path — which matches the observed 'DA3 traj fails' outcome whenever --da3-keyframes > 32. Fix: the first shared index is simply `s` (guaranteed in the previous chunk when overlap > 0): `shared_idx_in_prev = s`.

**Failure scenario:** reconstruct_with_da3(frames) with its own default n_keyframes=48 (or pipeline --da3-keyframes 48) on any clip: 2 chunks, identity alignment for chunk 2 -> folded/zigzag trajectory -> _da3_trajectory_plausible rejects it -> DA3 channel silently unusable above 32 keyframes, regardless of how well DA3 actually solved the poses.

### [MAJOR] src/ipm.py:84 — Camera height sign flipped: t=(0,+h,0) with +y-down places the camera BELOW the road — BEV samples the sky

In the documented OpenCV/vehicle convention (+y down), a camera h meters ABOVE the ground at y=0 must be at y=-h, but `t = [0, +camera_height_m, 0]` puts it at y=+h (underground). Every ground corner then projects ABOVE the horizon. Verified numerically with the module's own defaults (1280x720, hfov 70, h=1.4, pitch 6): the BEV homography samples image rows v in [-190, 234] while true ground points 4-40 m ahead project to v in [296, 576] (horizon ~264). So `warp_to_bev`/`stitch_bev_along_trajectory` build the 'synthetic satellite tile' entirely from off-frame black and sky/building pixels. tests/test_ipm.py passes because it only asserts non-emptiness of the warp (gray synthetic frames are non-zero everywhere). Fix: `t = np.array([0.0, -cal.camera_height_m, 0.0])` (the pitch matrix used as vehicle->camera is then correct for positive=down; note the ipm docstring calling R 'camera-to-vehicle' is also wrong — it is used as vehicle->camera).

**Failure scenario:** Run with --enable-ipm (cfg.enable_ipm): ipm_bev.png is a smeared sky/black canvas; step 8 then feeds it as `aerial_input` to the ORB aerial channel (pipeline.py:1365-1366), so the aerial ORB score is pure noise — the channel silently degrades ranking instead of helping it.

### [MAJOR] src/mapanything_trajectory.py:84 — stitch_windows Sim(3)/rigid fit is degenerate on collinear overlaps (straight driving) — rotation about the driving line is unconstrained

Consecutive windows are aligned using only the 3-8 shared camera POSITIONS. On a straight segment those points are collinear, so both `_umeyama` and the `_rigid` fallback leave the rotation component about the line direction completely unconstrained — SVD returns an arbitrary roll. The `scale_guard` only detects implausible SCALE, so a collinear overlap with plausible scale sails through and the new window enters the global frame with a random twist; every subsequent window inherits it. After `_positions_to_xy` PCA-flattens the 3D chain, turns that occur after a straight stretch project at distorted angles. Straight overlaps are the common case for driving (per project memory, the comma2k19 highway clip is exactly this regime, where MapAnything was 'worse end-to-end'). Fix: detect overlap collinearity (ratio of 2nd to 1st singular value of the centered overlap) and constrain the fit — e.g. solve for rotation using the shared frames' camera ORIENTATIONS (available from the model output) in addition to positions, or restrict to an in-plane (yaw-only) 2D fit.

**Failure scenario:** Route = 500 m straight then a 90-degree turn, window=16/step=8: the window chain along the straight section accumulates arbitrary roll at each collinear hand-off, so the turn's out-of-line points enter at a random orientation; after PCA projection the 90-degree turn renders as anything from ~30 to ~150 degrees -> bearing-signature match selects a wrong street geometry.

### [MAJOR] src/osm_data.py:152 — Node-based visited set makes self-crossing/looping true routes structurally unenumerable

_build_walk maintains `visited` as a set of NODES (line 144: `visited = {first_edge[0], first_edge[1]}`) and filters every out-edge with `e[1] not in visited` (line 152). Any driven route that passes through the same intersection twice — a loop back to the start, a figure-eight, or simply crossing a street it previously drove along at a shared OSM node — is truncated at the first revisit (the filtered `out` becomes empty and the walk breaks at line 156). If the truncation happens before 50% of target_length_m, walks_from_node discards the walk entirely (line 202 `length < 0.5 * target_length_m`). This directly violates candidate-pool completeness for the loop-shaped clips the project actually validates on (KITTI drive_0033 is a loop). **Failure scenario:** GT route is a 2 km loop returning through its starting intersection (KITTI 0033 pattern). Every enumeration from every start node on the loop stops at the node-revisit, yielding at best an open C-shaped 60-90% arc; if the revisit occurs before half the length prior, the walk is dropped altogether. The true closed route is never in the candidate pool, so no ranking channel (shape, turn, anchors, VPR) can ever select it — best-in-pool error is floored at the open-arc approximation.

### [MAJOR] src/osm_data.py:227 — First-edge cap [:3] plus sequential budget exhaustion starves 2nd/3rd start directions of multi-turn walks

walks_from_node iterates `for first in out0_sorted[:3]` — at a standard 4-way intersection of two-way streets (out-degree 4) the 4th direction is never enumerated from that node, and the order is by descending edge LENGTH, which has no relevance to route likelihood. ### Lens: selection-wall

#### [small] Un-gate the anchor/penalty fusion from enable_aerial_match and feed the consensus pick into anchor-primary

A second latent defect: W_AERIAL=0.5 forces the REFUTED appearance signal into every default fused score on homogeneous areas. - **Expected gain:** Correctness in all configs; on --no-aerial runs restores the full Ulm 491->236 m and Erbach 7 km->2.26 km anchor gains that are currently lost; on default runs removes a refuted 0.5-weight noise channel from the pick

#### [medium] Route-conditioned VPR likelihood: score each candidate walk against KartaView refs along ITS OWN path (hypothesis matched-filter, not retrieval)

The refuted direction was OPEN retrieval (per-frame top-1 anywhere in 3 km, then RANSAC track fit — 411 m noise, heading unusable). This inverts it into hypothesis EVALUATION, which is exactly what breaks a selection wall: for candidate c and query frame i we already know its map position (aligned_traj_xy[i]); score = mean over frames of max cosine similarity between the query embedding and cached ref embeddings within ~75 m of that position, minus the same statistic at random control positions (the control trick already validated the 91 m prior as real). No winner-take-all retrieval decision is ever made, so the 'look-alike street elsewhere' failure mode costs one frame's term, not the whole fit. All machinery exists: cached ref_imgs.npz/ref_meta.json + MegaLoc embedding code in src/kartaview_vpr.py and scripts/test_vpr_prior_accuracy.py (whose matrix shows the aggregate signal reaches 91 m to route). This is a per-candidate ABSOLUTE-appearance channel — the proven signal class — applied to the exact unrankable pool.

- **Expected gain:** If it ranks the true route first inside the VPR gate, Ulm headline goes from anchored-236 m to the ~91-160 m best-in-pool regime, and it becomes the missing selector on London/Karlsruhe where KartaView has coverage; it also replaces the crude centroid-distance vpr_penalty with per-route evidence
- **Risk:** Ref-density bias (walks through KartaView-dense streets score higher) — must normalize by ref count / control positions; MegaLoc look-alike confusion may leave parallel streets 50 m apart unseparated (then it still fixes neighbourhood-level mispicks like Erbach).
- **First experiment:** Offline, no pipeline change: load the cached Ulm pool from a GT run (result.json has gt_distance_m per candidate) plus the cached KartaView embeddings; compute the route-conditioned score for the top-40 pool and report Spearman(score, gt_distance) and the rank of the closest-to-GT candidate — directly comparable to scripts/bench_matching.py, which showed RMS has zero/negative correlation. Kill if correlation is ~0.

#### [small] Fix the along-track DOF of anchor-primary: slide the anchored route along its own path to maximize route-conditioned VPR likelihood

- **Expected gain:** Directly attacks start error, the worst headline metric (Ulm start 451 m, KITTI loop-phase ~724 m); plausibly halves start error wherever VPR refs exist
- **Risk:** Depends on idea #2's likelihood being informative along-track; on short KartaView-sparse routes the 1-D profile may be flat (then keep centroid placement as fallback — strictly no worse).
- **First experiment:** On Ulm cached data: take the current anchored candidate, sweep the centroid translation along the walk direction in 25 m steps +-400 m, score each offset with the route-conditioned likelihood, and evaluate GT start/mean error at the argmax vs the shipped centroid placement. #### [medium] OrienterNet as cross-hypothesis VERIFIER: compare fused belief likelihoods across the top-N distinct hypotheses

- **First experiment:** Ulm GT run with the gated pool: take the top-5 distinct hypotheses (one is near-true), run refine_route on ~10 keyframes per hypothesis using each walk polyline as prior+odometry, log the aligner's fused max log-prob. - **Expected gain:** Sharper vpr_penalty that penalizes wrong-orientation/wrong-progression candidates the centroid misses; plausibly moves the matcher pick itself (not just the translated pin) toward the true route on Ulm/Erbach, and adds selection power anywhere KartaView covers.

#### [small] Time-aware anchor scoring: distance at the SIGHTING time, not min-over-polyline
